# BLOCKING CONSTRAINTS — Read Before Anything Else

> These are not suggestions. Each constraint below was discovered through failure.
> Acknowledge each one explicitly before proceeding.

- [ ] CONSTRAINT: Sharp file I/O on Windows — direct `sharp(input).toFile(output)` with the same path, or `fs.copyFile` from a `.tmp` file immediately after `sharp.toFile()`, can produce `UNKNOWN` / `EPERM` errors. Always read the source into a buffer first (`fs.readFileSync`), then `sharp(buffer).webp(...).toBuffer()`, then `fs.writeFileSync`.

**Do not proceed until all boxes are checked.**

## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Sharp same-path write | Attempting to overwrite a WebP file directly with `sharp(file).toFile(file)` fails with "Cannot use same file for input and output". | blocking | Always use buffer-based pipeline for in-place optimization. |
| Windows temp-file rename race | `sharp.toFile(tmpFile)` may leave the file handle open long enough for `fs.renameSync` / `fs.copyFileSync` to fail with `EPERM`/`UNKNOWN`. | blocking | Read input to buffer, write output buffer to final path. |
| Forgetting leftover `.tmp` files | Temporary files created during failed rename attempts can be accidentally committed. | advisory | After any temp-file attempt, run `find public/images -name '*.tmp' -delete` and verify `git status`. |
| Including gallery in every deploy | `public/images/gallery` is ~36 MB and rarely changes; including it in `dist.zip` bloats deploys unnecessarily. | advisory | Build `dist.zip` from `dist/` excluding `images/` unless gallery files actually changed. |
